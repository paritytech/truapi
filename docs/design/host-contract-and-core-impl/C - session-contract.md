# C - Session / identity contract

> Part of the [host-contract & core-impl spec](<index.md>).

The core establishes the session by running pairing ([`request_login`](<B - core-impls.md>) /
[H](<H - sso-pairing-protocol.md>)), so it owns `SessionState`. This doc covers the `SessionInfo`
extension and core-owned persistence/restore.

## C1. Extend `SessionInfo` for a core-run session **(S-M)**

`SessionState` / `SessionInfo` live in `host_logic/session.rs`. Today:

```rust
pub struct SessionInfo { pub public_key: [u8; 32], pub lite_username: Option<String>, pub full_username: Option<String> }   // session.rs:15-23
pub struct SessionState { /* Mutex<{ current: Option<SessionInfo>, subscribers: Vec<UnboundedSender<..>> }> */ }            // session.rs:27-36
```

`SessionState`'s machinery is already right: `set_session` broadcasts `Connected` only on a
no-session->session transition; `clear_session` broadcasts `Disconnected` only if one was present;
`subscribe()` emits current-then-transitions; `current()` snapshots (session.rs:46-93). This drives
`connection_status_subscribe`(18) via the override at `runtime.rs:325-330`.

Because the core now runs pairing, `SessionInfo` carries the material the handshake produces
([H §3](<H - sso-pairing-protocol.md>)):

```rust
pub struct SessionInfo {
    pub public_key: [u8; 32],           // wallet root_user_account_id; the product-account derivation root
    pub ss_secret: [u8; 64],            // core's OWN expanded sr25519 statement-store secret (signs statement proofs)
    pub enc_secret: [u8; 32],           // core's P-256 ECDH private key (channel encryption)
    pub peer_enc_pubkey: [u8; 65],      // wallet persistent P-256 pubkey
    pub identity_account_id: [u8; 32],  // wallet identity pubkey
    pub session_id_own: [u8; 32],       // derived request/response topics
    pub session_id_peer: [u8; 32],
    pub lite_username: Option<String>,  // from on-chain identity, NOT the handshake
    pub full_username: Option<String>,
}
```

`ss_secret` is the core's own session statement-store key, **generated by the core** during pairing (the
wallet never sends a secret, [H](<H - sso-pairing-protocol.md>)); it signs the core's statements and the
product statement proofs ([B create_proof](<B - core-impls.md>)). None of these secrets leave the core.
Account-only injection APIs such as `set_active_session(pubkey, lite, full)` are not part of this plan:
they cannot rehydrate `ss_secret`, ECDH material, peer keys, or derived session ids. Real v1 restore uses
the core-owned persisted session blob in C2. Usernames are filled by a separate on-chain People-chain
identity lookup keyed by `public_key`.

## C2. Persistence + restore **(M)**

```
  request_login succeeds                         core init (cold start)
        |                                              |
        v                                              v
  SessionState.set_session(info)                 SessionStore.subscribe()
        |  Connected broadcast                          |  first tick is current-state signal
        v                                              v
  SessionStore.write(encode(info))               SessionStore.read()
        |                                             Some(blob) -> decode -> set/replace session
        v                                             None       -> clear_session / stay Disconnected
  (new successful pairing replaces old blob)     future ticks repeat read/decode
        ...
  logout: clear_session + SessionStore.clear()
```

The core writes opaque encoded bytes for the full `SessionInfo` via a dedicated host/global
`SessionStore` capability on establish, and subscribes to the store on init to re-hydrate `SessionState`.
Do **not** reuse the existing `Storage` trait: that is product-scoped local storage semantics, while the
wallet session is host-global, secret-bearing, and shared across product runtimes. The host does not see
typed session fields; Rust owns the schema, versioning, migration, and validation. V1 is **single-session
only**: `SessionStore` exposes at most one blob and a new successful pairing replaces the old blob.
If the blob is missing, unreadable, fails version/schema validation, or fails decode, the core
clears `SessionStore`, logs/emits diagnostics, and stays `Disconnected`; it must not start with a partial
session.
Rust does not encrypt or MAC the persisted blob itself; confidentiality and tamper resistance are the
host storage layer's responsibility. The core still validates the bytes before restoring a session.
For dotli web, back `SessionStore` with the same shared host-origin storage path used by current auth
(`host.<root-domain>` localStorage via the protocol iframe), but use a new Rust-owned key/prefix such as
`TRUAPI_SESSION_<siteId>`. Do not reuse `PAPP_*` or `SsoSessions`; those names imply host-papp's
multi-session `Vec<Session>` encoding.
`SessionStore.subscribe()` emits one tick immediately, then future change ticks for both same-runtime
writes/clears and cross-tab/process changes. The notification carries no session bytes; after each tick the
core calls `read()` and validates the current blob. That avoids a read-then-subscribe race and keeps every
runtime on the same path: a clear yields `None`, clears in-memory `SessionState`, and broadcasts
`Disconnected`; a valid replacement blob is decoded, replaces in-memory `SessionState`, and broadcasts the
normal connection transition. The core should dedupe equivalent blobs/session ids before rebroadcasting if
the host emits after a local write that the core already applied. dotli web should implement this with the
same local emit plus protocol-iframe BroadcastChannel-style shared-storage notifications it uses today.
The persisted format is **core-defined**; the cutover from `@novasamatech` sessions intentionally
requires a one-time re-pair ([E5](<E - open-questions.md>)). Logout clears both in-memory state and the
persisted blob through the same core-owned session store.

## C3. Public logout / disconnect API **(S)**

dotli UI needs a direct host->core logout action. Expose a public core API such as `disconnect()` /
`logout()` through WASM, UniFFI, and the JS worker package. Hosts should call this API, not
`SessionStore.clear()` directly.

Core behavior:

- best-effort send SSO session-channel `disconnected` over the People-chain statement-store protocol if
  the session channel is currently usable; do not wait indefinitely and do not fail logout if this send
  fails;
- stop/tear down active session-channel subscriptions and pending request waiters; in-flight
  request/response operations must resolve as failures rather than hanging. Current dotli delegates this
  to `host-papp`'s session manager (`adapter.sessions.disconnect(session)`), whose debug contract reports
  channel teardown as `session:host_action_failed`; the Rust core should make that lifecycle explicit.
- clear in-memory `SessionState` and broadcast `Disconnected`;
- clear the single persisted session blob through `SessionStore.clear()`;
- tolerate repeated logout when already disconnected.

Peer-originated `disconnected` messages over the SSO session channel route through the same internal
cleanup path, including `SessionStore.clear()`. Do not keep a persisted session after the SSO peer says the
session is disconnected.

Current host-papp has a 180s session request-queue timeout for `signPayload`, `signRaw`,
`createTransaction`, and `requestResourceAllocation`; dotli's signing/transaction UI wrappers add a 300s
outer fallback after the user confirms the modal. `getRingVrfAlias` does not add a separate timeout in the
current checkout. Preserve the same user-visible failure shape in Rust: peer decline maps to the method's
domain rejection/unknown path, transport teardown maps to `Unknown { reason }` where the host-api type
lacks a dedicated disconnected variant, and no operation waits forever after local or peer disconnect.
